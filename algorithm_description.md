# Краткое описание алгоритма

Аномалия ищется на уровне триггера `(SubjectID, researchdate, BrandID, CategoryDelivery)`. Для каждой такой группы среди строк `BrandinDelivery = 1` и непустого `CategoryDelivery` считается:

```text
daily_ots = Weight(SubjectID, researchdate) * count_rows(SubjectID, BrandID, CategoryDelivery, researchdate)
```

Удаление выполняется на уровне `(SubjectID, researchdate)`: если хотя бы один бренд стал триггером, из очищенной выборки удаляются все строки этого респондента за этот день.

## Score и порог

Метод ориентирован только на чрезмерно высокие значения OTS. Для устойчивости к тяжелым хвостам используется `log1p(daily_ots)`. Внутри пары `(CategoryDelivery, BrandID)` рассчитываются медиана и MAD; score выводится как отношение `daily_ots / threshold`.

Порог:

```text
threshold_log = median + 3.5 * 1.4826 * MAD
threshold = exp(threshold_log) - 1
```

Коэффициент `3.5` соответствует стандартному правилу modified z-score для outlier detection. Для редких брендов с числом наблюдений меньше 20 брендовый порог не используется; вместо него берется fallback на `CategoryDelivery` и глобальное распределение. Итоговый порог для строки - максимум из доступных брендового, категорийного и глобального порогов. Это защищает от ложных срабатываний на малых выборках и редких брендах.

Строка считается аномалией, если ее `log1p(daily_ots)` выше итогового порога. В `anomaly_reasons.csv` записываются `daily_ots`, `score`, `threshold` и текстовая причина с источником порога.

## Почему подход разумен

- Алгоритм работает на уровне бренда, а не только категории.
- Порог рассчитывается из данных и не использует hardcode `SubjectID`, дат, брендов или целевого процента удалений.
- Медиана и MAD устойчивы к экстремальным значениям, поэтому сами пики меньше влияют на порог.
- Логарифмирование снижает влияние тяжелого правого хвоста OTS, но сохраняет порядок значений.
- Fallback на категорию и глобальное распределение решает проблему малых выборок.
- Решение детерминировано и не использует случайные модели.
